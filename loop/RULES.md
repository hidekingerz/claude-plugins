# RULES — エージェントが絶対にやってはいけないこと

> VERIFY の品質ゲートと並ぶガードレール。PLAN / EXECUTE でこのルールを最優先で守り、抵触する
> 場合は変更しない。

## 禁止事項（Hard rules）

- 保護ブランチ（`main`/`master`）へ直接コミット・push しない。作業は必ず feature ブランチで行う
  （現在のブランチがそれか DISCOVER で確認。違えば何も変更せず Open に記録して終了）。
- **PR を自動作成・自動マージしない。** このループの責務は「修正 → コミット → push → issue コメント」
  まで。PR 化とマージは人間。
- `auto-fix` ラベルの付いていない issue を勝手に修正しない。
- `loop-wip` / `loop-needs-human` が付いた issue を選ばない（着手中/人間待ち）。
- issue の受け入れ条件を超える変更をしない（無関係リファクタ・大規模横断変更・仕様変更）。
  1周 = 1 issue = 最小変更。
- VERIFY（`./loop/verify.sh`）が通っていない変更をコミットしない。
- 既存のスキル/エージェント定義を「検査を通すために」内容を空にする・骨抜きにするような
  ごまかしをしない。frontmatter を形だけ埋めて中身を壊さない。
- `git commit --amend` や同一 issue の追加コミットをしない（1 issue = 1コミット）。
- 秘密情報・認証情報（トークン等）をコード・ログ・コミット・issue コメントに書かない。
- `loop/` 配下（ループ設定）を勝手に書き換えない（`loop/MEMORY.md` への追記は除く。MEMORY は
  毎周更新する）。`loop/verify.sh` 自体を緩めて検査をすり抜けない。
- issue を **close しない**（close は人間が PR マージ時に行う）。

## 判断保留（自動修正せず人間に回すべき issue）

次に当てはまる issue は修正を試みず `loop-needs-human` を付けて次へ:

- 受け入れ条件が曖昧・未確定、または解釈が割れるもの。
- 破壊的変更（公開スキル/プラグインの互換を壊す、ファイル削除・大規模移動を伴う）。
- 設計判断・トレードオフの選択を要するもの。
- 静的ゲートでは正しさを一切確認できず、リスクが高いもの。

## 進め方の原則

- これは自律実行ループ。`brainstorming`/`writing-plans` 等のメタ/プロセス系スキルは起動せず、
  `LOOP_PROMPT.md` の5段階を直接実行する。
- 判断に迷ったら**安全側（変更しない / 人間に回す）**に倒し、`loop/MEMORY.md` の Open に判断ポイントを記録する。
- 「VERIFY 緑」≠「issue 解決」。issue の要求そのものを満たしたかを客観判定する。
- 着手可能な issue が残る限り `LOOP_DONE` を出さない。残りが全部 `loop-needs-human` なら停止してよい。
