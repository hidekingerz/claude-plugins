# ゲート設計と運用の指針（実戦の教訓）

このループの成否は **VERIFY ゲート（と完了ゲート）の設計**でほぼ決まる。ある実プロジェクト
（Tauri→Deno Desktop 移行、自律ループで M1〜M8 を完走）で得た教訓を凝縮する。

## 1. ゲートがループそのもの（最重要）

ループは「DoD」ではなく「**ゲート**」に最適化する。**ゲートが観測できない失敗モードでは、ループは
自信を持って "green のまま壊れたもの" を出荷する。**

- 実例: ビルド + 単体テストの VERIFY が緑だったので `LOOP_DONE` が出たが、アプリは起動すると
  **白画面**だった。VERIFY がフロントの**実行時**を一切観測していなかったのが原因。
- 対策: **VERIFY/完了ゲートを「本当の DoD」に一致させる。** GUI/ランタイムを伴うなら「ビルド+単体」
  では足りない。**ランタイム/GUI smoke**（ヘッドレス起動して実行時エラーを検知）を完了ゲートに入れる。
- 下記の実例は GUI 移行（Tauri→Deno）だが、原則は型を問わない。**lib/CLI** では「実行時の観測」は
  GUI smoke ではなく**結合/振る舞いテスト + 公開 API の静的ゲート**（例: 公開関数が export されて
  いる・禁止 import が無いの grep）になる。要は「DoD で謳ったことを観測するゲート」を必ず置く。

### ランタイム/GUI smoke の作り方（汎用パターン）

「起動して実行時エラーが出ないこと」をヘッドレスで観測する小さなテストを用意する。例:

- アプリを起動し、未捕捉エラー（`window.onerror` / `unhandledrejection` / コンソールエラー）を
  プロセスの stderr へ転送 → ログに出たら fail。
- HTTP サーバ型なら起動して主要ルートを叩き、非2xx/エラーレスポンスで fail。
- デスクトップ/ブラウザ型なら、ビルド→ヘッドレス起動→数秒待って `[error]` 行が無いことを確認。
  （バンドルアプリで親ターミナルに stderr が出ない場合、OS の仕組みで stderr をファイルへ
  リダイレクトして捕捉する等の工夫が要る。）

### MCP ツールによる検証（GUI/ランタイムの強力な観測手段）

bash の smoke より踏み込んで、**MCP ツールで実アプリを操作して検証**できる。例:

- ブラウザ/Web アプリ: Playwright MCP（`browser_navigate` / `browser_snapshot` /
  `browser_console_messages` / `browser_click`）や chrome-devtools MCP で、実際に画面を開き
  操作し、**コンソールエラー・ネットワーク失敗・要素の有無**を観測する。「白画面」「ボタンが
  効かない」のような bash では捉えにくい失敗を直接掴める。検証は「開いて見る」で終えず、
  **①変更した UI を操作して状態変化を確認 ②before/after スクリーンショット ③コンソール
  エラー/警告 0 件 ④（性能が DoD なら）パフォーマンストレースで Core Web Vitals** まで行う。
  実例: `npx playwright test`（コンソール監視付き）では検出できない favicon 404 を、
  MCP の実ブラウザ検証がコンソールエラーとして捕捉→修正できた。
- その他、DB/API 系 MCP で投入後の状態を検証する等。

**重要な制約と組み込み方:**

- `run.sh` の `VERIFY_CMD` は **bash 専用**（`bash -c`）。MCP ツールはそこから直接は呼べない。
- MCP 検証は **エージェント（`claude -p`）自身に行わせる。** `claude -p` は設定済み MCP サーバへ
  アクセスできるので、**LOOP_PROMPT の VERIFY 段／VISION の完了ゲート**に「MCP で〜を検証する」と
  明記すれば、エージェントがそのツールを使って自己検査する。
- bash の `VERIFY_CMD` に乗せたいなら、MCP 相当を **CLI でラップ**する（例: Playwright なら
  `npx playwright test` のような CLI テスト）。MCP そのものは run.sh からは動かない。
- **ヘッドレス/無人実行の注意:** 対話認証が要る MCP サーバ（例: ログインが必要なもの）は
  headless な `claude -p` では使えないことがある。Playwright/chrome-devtools のような
  認証不要のものは無人ループでも使える。

使い分け: **毎周の VERIFY**は速い bash 静的検査に留め、**MCP での重い GUI 検証は完了ゲート**
（LOOP_DONE 前にエージェントが実施）に置くのが基本。

## 2. 毎周の VERIFY と「完了ゲート」を分ける

重い/全体的なゲート（smoke・「禁止 import が残っていないか」の grep ゲート等）を**毎周の VERIFY に
入れない**。段階移行の途中状態では赤になり、ループがコミットできず進まなくなる。

- **毎周の VERIFY** = 速い静的検査（fmt/lint/typecheck/unit）。途中状態でも緑にできる。
- **完了ゲート**（LOOP_DONE の条件）= VERIFY + grep ゲート + smoke + 結合テスト。全緑で初めて完了。
- VISION.md の「完了ゲート」節にこの2層を明記する。

## 3. ループは VISION の誤りも忠実に実装する

ループは仕様を疑わない。VISION/RULES が誤っていると、ループは**正しくその誤りを実装する**。
- 実例: VISION が「コンポーネントは無改変」と書いたため、ループはコンポーネント内の移行漏れを
  正しく放置し、それが実行時バグ（白画面）になった。
- 対策: VISION の DoD・スコープを**実際の完成条件**と突き合わせて書く。盲点を作らない。

## 4. 節目で人間検証に「止まる」設計に

GUI の「動く/操作できる」「体感速度」「位置がカーソルに合う」等は smoke でも判定しづらい。
これらは**ループに判定させず、節目で人間の目視確認**に委ねる。VISION の注意書きにその旨を残す。
- 実例: ビューワー起動・メニュー位置・ウィンドウ閉じる挙動・特定形式の遅延は、すべて人間テストで
  発覚した（ループの自動ゲートでは見えなかった）。

## 5. 運用ハードニング（同梱 run.sh に実装済み）

地味な配管が成否を分ける。本スキルの `run.sh` は以下を実装済み:

- **堅牢な LOOP_DONE 検出**: 行全体がマーカーと一致する場合のみ停止。エージェントが説明文中で
  `LOOP_DONE` に言及しただけで誤停止する事故を防ぐ（`grep -q` 直当ては危険）。
- **連続失敗サーキットブレーカ**: `MAX_CONSEC_FAIL`（既定3）連続で agent 起動失敗したら停止。
  **session/API 制限**で iteration が大量空回りするのを防ぐ。
- **保護ブランチガード**: `main`/`master` 直走を拒否。
- **VERIFY 空の警告**: 品質ゲート無し運用を警告。

LOOP_PROMPT は **1タスク=1コミット・amend 禁止・ハッシュを MEMORY に書かない**を指示
（コミットノイズ防止）。

### 実走で追加で学んだ落とし穴（配管）

- **マーカーの "うっかり単独行出力"**: 行アンカー検出は堅牢だが、エージェントが「まだ未完了」と本文で
  書きつつ、別行にマーカーだけを出力して**誤完了停止**した実例がある。DoD 未達なのに停止するので
  被害は大きい。対策は検出側ではなく指示側: `LOOP_PROMPT.md` に「未完了のうちはマーカーを独立行で
  出さない／言及時は同じ行に他の語を続ける」を明記し、`DONE_MARKER` をプロジェクト固有名に寄せる。
- **VERIFY のインストールは `npm install` でなく `npm ci`**: host と container の npm バージョン差で、
  `npm install` は `package-lock.json` を毎周わずかに書き換える（例: npm 11 が書く `libc` フィールドを
  npm 10 が剥がす）。作業ツリーが汚れて 1タスク=1コミットに混入しかねない。`npm ci` は lockfile を
  書き換えず node_modules をクリーンに再構築するので churn ゼロ。インストールガードは
  `[ -x node_modules/.bin/<tool> ] || npm ci; <lint/typecheck/test>` の形が安定。
- **孤立コンテナ**: `run-in-docker.sh` のラッパーが殺されても（Ctrl-C・実行時間上限）、コンテナは
  detach したまま監視外で回り続けうる。同梱 `run-in-docker.sh` は `--name` 付与＋ `trap docker stop`
  で INT/TERM・正常終了時に停止する（SIGKILL は trap 不能なので、その場合は名前で `docker stop`）。

## 6. session/API 制限とコスト

ループは「メインの監視セッション + 毎周の `claude -p`（各々フルエージェント）」が**同じ契約の枠**を
消費する＝**使用量の増幅器**。連続バッチ + 高 effort は容易に上限に当たる。
- 対策: バッチを小さく区切る／`--effort` は必要な周だけ／ブレーカで空回りを早期停止／上限と
  照らしてループ規模（周数・effort・並行度）を設計する。
- **モデル・ルーティング**: 定型的な反復（EXECUTE 中心の周）は小さく速いモデルで足りることが多い。
  `AGENT_CMD='claude -p --model claude-sonnet-5 --dangerously-skip-permissions -'` のように周のモデルを
  下げ、**判断が重い所（checker・設計判断・難所の周）にだけ最上位モデル**を使うと総コストが大きく下がる。
- **パイロット実行**: いきなり本番規模で回さず、`MAX_ITER=2〜3` で1〜2周だけ回して1周あたりの
  消費と挙動（どこで詰まる/踏み越えるか）を実測してから本走する。
- **消費の可視化**: 対話セッション側では `/usage` がスキル/サブエージェント/MCP 別の内訳を出す。

## 7. 未コミット作業の救出

session 制限が iteration を途中で殺すと、VERIFY を通る完成済みの作業が**未コミットで残る**ことが
ある。再走前に `git status` を確認し、VERIFY が緑なら**救出コミット**する（捨てて再実行は無駄）。

## 8. 無人実行の権限とサンドボックス（安全網の実効範囲を誤解しない）

無人ループでは `claude -p --dangerously-skip-permissions -` が現実的。これは**権限確認を
一切バイパスする**＝エージェントが確認なしに任意のシェルコマンドを実行できる、ということ。

### ★安全網が「実際に何を守るか」を取り違えない（最重要）

このループの4つの安全網は**それぞれ守る対象が違う**。過信は事故のもと:

| 安全網 | 実際に守る範囲 | 守らない範囲 |
|---|---|---|
| ① feature ブランチ隔離 | git コミット履歴の分離だけ | ファイルシステム全域の書き換え/削除、リポジトリ外への操作 |
| ② RULES.md | **プロンプトレベルの努力目標**（LLM は無視・誤読しうる。強制力ゼロ） | 実行時の一切 |
| ③ VERIFY ゲート | 変更を**コミットしてよいか**の判定 | エージェントが既に実行した任意コマンド（VERIFY は事後） |
| ④ MAX_ITER / ブレーカ | 反復の暴走・API 空回りの停止 | 1周内での破壊的操作 |

つまり**①〜④は「git のコミット」と「ループの暴走」を守るだけで、ファイルシステム・秘密情報・
ネットワークは一切守らない。** `--dangerously-skip-permissions` 下のエージェントは cwd こそ
リポジトリルートだが、シェル権限は**実行ユーザー全体**に及ぶ。原理的に `~/.ssh`・
`~/.aws/credentials`・`.env` の読み取り、ホームの書き換え、外部ネットワーク送信まで可能。

さらに危険なのが**プロンプトインジェクション**。DISCOVER 段でエージェントはリポジトリ内の
コード・テスト・依存パッケージの README やフィクスチャ等の**信頼できないテキスト**を読む。そこに
指示が仕込まれていると、無人・権限バイパスのエージェントがそれに従いうる（私的データ＋外部通信＋
信頼できない入力が揃った "lethal trifecta" が全部成立する構成）。**RULES.md はこれを止められない。**

### 対策: ホスト環境そのものを隔離する

git ブランチはホストを守らない。守るのは**実行環境の隔離**:

- **使い捨てコンテナ / VM / 専用低権限ユーザーで実行する。** リポジトリだけをマウントし、ホームの
  機密パス（`~/.ssh`, `~/.aws`, `~/.config/gh` 等）はマウントしない。同梱の Docker ランナー
  （`assets/Dockerfile` + `assets/run-in-docker.sh`）を使うのが最短。手順は `SKILL.md` の
  「Docker 隔離実行」節。
- **コンテナ隔離にも残留物の穴がある**: rw マウントしたリポジトリの `.git/hooks`・`.git/config`・
  `loop/*.sh` を改変されると、**次にホストで git やループを実行した時にホスト側でコードが走る**
  （コンテナからホストへの持続化ベクタ）。同梱ランナーは実行前後のハッシュ比較で警告する（検知で
  あって防止ではない）。警告が出たら差分確認まで ホストで git を叩かない。
- **ネットワークを絞る。** 原則遮断し、必要な宛先（パッケージレジストリ・API エンドポイント）だけ
  許可する。無制限のアウトバウンドは漏洩経路。
- **秘密情報を実行環境から除去する。** ループが要らない資格情報は環境変数・ファイルごと渡さない。
  RULES.md の「秘密情報を書かない」はプロンプト指示にすぎず強制力が無い。
- **VERIFY_CMD もリポジトリ由来コードを実行する信頼境界の一部**と心得る。`npm test` 等は
  `package.json` スクリプトや依存の postinstall を走らせる＝悪意/改変されたリポジトリでは無人の
  コード実行経路になる。隔離環境で回す理由はここにもある。
- 適用先が**信頼できないコード**（未知の OSS 等）なら、`--dangerously-skip-permissions` を使わない
  （権限確認を残す=無人にしない）選択肢も検討する。
- 保護ブランチはサーバ側/フックで別途守る（これは④とは別レイヤ）。

## 9. maker/checker 分離 — フレッシュ文脈のセカンドエージェントにレビューさせる

作った本人（maker）は自己採点が甘くなりがちで、**テストを緑にする過剰適合**（テスト入力への
ハードコード分岐等）は VERIFY（テスト実行）では原理的に検出できない。**maker と文脈を共有しない
別エージェント（checker）にコード/diff だけを見せてレビューさせる**と、この種の失敗を捕捉できる
（機構検証で、テストは全緑のハードコード実装を checker が REJECT した実績あり）。

組み込み方（★checker は maker の外＝**オペレータ/ハーネス層**で実行する）:

- **エージェント自身に checker を起動させない。** 理由は2つ（どちらも full-loop 検証で確認済み）:
  1. **認証が通らない**: Claude Code は自身の資格情報（`ANTHROPIC_API_KEY` /
     `CLAUDE_CODE_OAUTH_TOKEN`）を **Bash 子プロセスの環境から除去する**（インジェクション経由の
     持ち出し対策）。コンテナ env にトークンがあり外側の `claude -p` は認証できていても、
     エージェントが実行する入れ子の `claude -p` は `Not logged in` で失敗する（実測）。
  2. **信頼構造**: maker が自分の checker を起動して結果を報告する構図では、checker の判定も
     ループの自己申告に含まれてしまう（偽装・省略しうる）。分離の意味が薄れる。
  なお VISION の完了ゲートに「エージェントが checker を実行し APPROVE を得る」と書くと、認証不能に
  より**恒久ブロック**する（実測ではエージェントは誠実に `LOOP_DONE` を保留し MAX_ITER まで到達＝
  誤完了はしないが前進もしない）。checker をゲートに書くなら「オペレータが実施する」ものとして書く。
- **ループ停止後〜マージ前に、オペレータ（またはループ外のラッパースクリプト）が実行する**:

  ```bash
  git show <featコミット> | claude -p --model claude-opus-4-8 \
    "あなたは checker（レビュアー）です。この diff をレビューし、テスト入力への過剰適合・
     仕様との乖離・未処理エッジケースを探せ。結論を APPROVE / REJECT(理由) の1行で終えること。" -
  ```

  REJECT なら理由を `loop/MEMORY.md` の Open に `[operator/...]` 注記で転記して再走する。
- 対話セッションで節目に確認するなら、組み込みの **`/code-review`** がそのまま使える
  （フレッシュ文脈のレビューエージェントが現在の diff を検査する）。
- 要点は **maker と checker の役割プロンプトを分ける**こと。同一プロンプトの2回実行では同じ盲点を
  共有しやすい。checker には「粗探し（refute）する役」だと明示する。

## オペレータの主戦場

ループ化しても人間の仕事は消えず、**シフトする**:「作業する人」→「**ゲートを実 DoD に一致させ、
ゲートが見えない所を検証し、MEMORY 注記で方向付けする人**」。ゲート設計こそが人間の主戦場。
