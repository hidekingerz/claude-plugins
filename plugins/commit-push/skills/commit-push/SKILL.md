# Commit & Push スキル

## 概要

作業中の変更をコミットし、リモートリポジトリへ push する。

---

## ステップ

### 1. 現在の状態を確認する

以下を並列で実行する：

- `git status` — 変更ファイルと未追跡ファイルを確認
- `git diff` — staged / unstaged の差分を確認
- `git log --oneline -5` — 直近のコミットメッセージのスタイルを把握

### 2. コミットメッセージを決定する

- 引数が指定されている場合はそれを使う
- 指定がない場合は差分から適切なメッセージを生成する
- リポジトリのコミット履歴のスタイルに合わせる
- 末尾に必ず以下の Co-Authored-By を付与する（モデル名は実際に使用しているモデルに合わせる）：
  ```
  Co-Authored-By: Claude <モデル名> <noreply@anthropic.com>
  ```
  例: `Claude Sonnet 4.6`, `Claude Opus 4.6`, `Claude Haiku 4.5` など

### 3. ユーザーにコミット内容を確認する

コミットを実行する前に、以下の形式でユーザーに提示し、承認を得る：

```
以下のファイルをコミットします。よろしいですか？

コミット対象ファイル:
- <ファイル1>：<変更のサマリ>
- <ファイル2>：<変更のサマリ>
...

コミットメッセージ:
<メッセージ>
```

ユーザーが否定・キャンセルした場合はそこで中断する。

### 4. ファイルをステージングする

- `.env` や認証情報など機密ファイルが含まれていないか確認する
- `git add` で関連ファイルを個別に指定する（`git add -A` は避ける）

### 5. コミットする

HEREDOC を使ってメッセージを渡す：

```bash
git commit -m "$(cat <<'EOF'
<コミットメッセージ>

Co-Authored-By: Claude <モデル名> <noreply@anthropic.com>
EOF
)"
```

### 6. ユーザーに push を確認する

push を実行する前に、以下の形式でユーザーに確認を求める：

```
<ブランチ名> を origin へ push します。よろしいですか？
```

ユーザーが否定・キャンセルした場合はそこで中断する。

### 7. Push する

```bash
git push
```

### 8. 完了を確認する

`git status` を実行して push が成功したことを確認する。

---

## 注意事項

- `--force` push は絶対に行わない（ユーザーが明示的に指示した場合のみ）
- `--no-verify` は使用しない
- main / master ブランチへの force push は拒否する
- 機密ファイルはコミットしない
