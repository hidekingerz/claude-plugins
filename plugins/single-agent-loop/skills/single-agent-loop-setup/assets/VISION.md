> 全項目が検証可能でグリーンになるまで `LOOP_DONE` を出力しない。曖昧だとループは終わらない。

## 完了の定義（Definition of Done）— 上から順に進める・検証可能な箇条書きで

> ★最重要: DoD は「**本当に完成した状態**」を書く。ビルド/単体テストが緑なだけでは不十分なら、
> 「実際に動作する」ことまで DoD に含め、それを観測できるゲート（下記）を用意する。

- [ ] {{例（共通）: 機能 A が実装され、対応する単体テストが pass する}}
- [ ] {{例（共通）: 既存の挙動と等価（リグレッションが無い）}}
- [ ] {{例（lib/CLI）: 全公開関数/コマンドに単体テストがある／公開 API のシグネチャが意図通り}}
- [ ] {{例（GUI/アプリ）: 実際に起動し主要操作ができる（= ランタイム/GUI smoke が緑）。GUI 無しなら削除}}

## 完了ゲート（LOOP_DONE を出してよい条件）

DoD を満たし、かつ次がすべてグリーンであること（**プロジェクト型に応じて該当する行だけ残す**）:

- [ ] {{例（共通）: 毎周の VERIFY が緑（例: `npm run lint && npm run typecheck && npm test`）}}
- [ ] {{例（lib/CLI 向け静的ゲート）: 「全公開関数が export されている」「禁止 import が無い」等の grep ゲート}}
- [ ] {{例（GUI/アプリのみ）: ランタイム/GUI smoke（ヘッドレス起動して実行時エラーが出ない）}}
- [ ] {{例（GUI/アプリのみ）: MCP での GUI 検証。Playwright/chrome-devtools MCP で実際に画面を開き、
      ①変更した UI を操作して期待どおり状態が変わる ②before/after のスクリーンショットを撮る
      ③コンソールエラー/警告が 0 件 ④（性能が DoD なら）chrome-devtools MCP でパフォーマンス
      トレースを取り Core Web Vitals を確認 — まで実施して初めて緑。エージェントが VERIFY 段で
      実施する（run.sh の bash VERIFY_CMD では直接呼べない）。GUI 無しなら削除}}

> ★毎周の VERIFY は速い静的検査（fmt/lint/check/test）に留め、重いゲート（smoke 等）は
> **完了判定にのみ**使う。さもないと段階移行の途中でコミットできず進まない。

## 進行上の注意

- 「ビルド成功」≠「完了」。VERIFY が緑でも実行時に壊れていることがある。完了ゲートが全緑になって
  初めて `LOOP_DONE`。
- GUI/実行時の「動く/操作できる」の最終確認が自動化困難な場合は、その旨を Open に残し、節目で
  **人間の目視確認**に委ねる（ループに判定させない）。
